# laundry-monitor

Bare-metal Rust application for the ESP32s microcontroller. Uses an SW-420 vibration sensor to detect when a laundry cycle has completed. When the sensor has been still for a configurable idle timeout, the device sends an HTTP POST notification to a locally-running ntfy instance over WiFi.

Reference: https://esp32.implrust.com

---

## Current state (start of next iteration)

The device is fully functional end-to-end:

- Connects to WiFi via `esp-radio 0.17`
- Acquires a DHCP lease via `blocking-network-stack` + smoltcp 0.12
- Polls GPIO25 every 5 ms for SW-420 vibration
- Sends an HTTP/1.0 POST to a local ntfy container when the sensor has been idle for `IDLE_TIMEOUT`
- `IDLE_TIMEOUT` is currently **30 seconds** (for testing — raise to 300 s for production)

### Architecture notes

- `esp-radio 0.17` `WifiDevice` implements `embassy_net_driver::Driver`, **not** `smoltcp::phy::Device`. A custom `WifiAdapter` bridges the two traits using a noop `Waker` (blocking poll loop needs no async wake).
- `blocking-network-stack` discovers the DHCP socket by scanning the `SocketSet` passed to `Stack::new()`. **A `smoltcp::socket::dhcpv4::Socket::new()` must be pre-added to the socket set before constructing `Stack`.** Without it, `dhcp_socket_handle` stays `None` and DHCP never runs.
- `static mut` socket buffers (`SOCK_RX`, `SOCK_TX`) are accessed via `addr_of_mut!` to satisfy Rust 2024's `static_mut_refs` lint.
- WiFi credentials are supplied via environment variables (`WIFI_SSID`, `WIFI_PASSWORD`) at build time.
- ntfy connection target (`NTFY_HOST`, `NTFY_PORT`, `NTFY_TOPIC`) are constants in `src/bin/main.rs`.

## Hardware

| Component | Detail |
|-----------|--------|
| MCU | ESP32s (ESP32 DevKit V1) |
| Sensor | SW-420 vibration sensor module |

### Wiring

| SW-420 Pin | ESP32 Pin |
|------------|-----------|
| VCC | 3.3V |
| GND | GND |
| D0 | GPIO25 |

**Notes:**
- All three connections (3V3, GND, GPIO25) are on the same side of this ESP32s board.
- The SW-420 on this module is **active-high**: D0 goes HIGH when vibrating, LOW when still.
- The red LED on the sensor module flashes when vibration is detected.

## Key implementation decisions

| Decision | Value | Reason |
|----------|-------|--------|
| Poll interval | 5 ms | SW-420 produces brief pulses; 100 ms misses them |
| Idle timeout | 30 s (testing) / 300 s (production) | Configurable via `IDLE_TIMEOUT` |
| Pull resistor | `Pull::Down` | Sensor rests LOW; pull-down stabilises idle reads |
| GPIO logic | `is_high()` | Sensor is active-high on this module |
| Network | `blocking-network-stack` + smoltcp | No async executor needed; busy-wait polling fits the single-task model |
| HTTP | HTTP/1.0 POST | Server closes connection after response; no header parsing needed |
| Logging | `esp_println::println!` | Plain UART output readable by `espflash --monitor` |

## Dependencies

```toml
esp-hal             = { version = "~1.0",  features = ["esp32", "unstable"] }
esp-bootloader-esp-idf = { version = "0.4.0", features = ["esp32"] }
esp-println         = { version = "0.13",  features = ["esp32"] }
esp-alloc           = "0.9"
esp-radio           = { version = "0.17.0", features = ["esp32", "wifi"] }
embassy-net-driver  = "0.2"
esp-rtos            = { version = "0.2",   features = ["esp32", "esp-radio"] }
blocking-network-stack = { git = "...", features = ["dhcpv4", "tcp"] }
embedded-io         = "0.6"
heapless            = "0.9"
smoltcp             = { version = "0.12",  features = ["medium-ethernet", "proto-ipv4", "socket-tcp", "socket-dhcpv4"] }
```

Key constraints:
- `esp-hal` **must** have the `"unstable"` feature — required by esp-radio and esp-rtos.
- `esp-alloc 0.9` uses `heap_allocator!(size: N)` macro syntax (not a struct method).
- `Rng::new()` takes no arguments in esp-hal 1.0.
- `smoltcp` must be an explicit dependency (not just transitive) because we use it directly.

## Toolchain setup

Requires the Xtensa GCC toolchain on PATH. If not already sourced:

```bash
espup install          # first time only
source ~/export-esp.sh
```

---

## Build & flash

```bash
export WIFI_SSID="your-network"
export WIFI_PASSWORD="your-password"
cargo build            # compile only
cargo run              # compile, flash, and open serial monitor
```

The runner in `.cargo/config.toml` automatically calls `espflash flash --monitor --chip esp32`.

---

## ntfy Docker container

```bash
cd docker/
docker compose up -d   # starts ntfy on http://localhost:8080
```

Subscribe via browser or the ntfy app to topic `laundry-monitor-test` on your local machine's IP.
